# Adapter / Facade / Proxy: коли застосовувати

## Вступ

Патерни Adapter, Facade та Proxy використовуються для інтеграції систем, спрощення роботи з ними та контролю доступу до об’єктів.  
Хоча вони мають схожу структуру (часто працюють через посередника), їхнє призначення різне.

---

# 1. Adapter (Адаптер)

## Коли застосовувати

Adapter використовують тоді, коли потрібно поєднати два несумісні інтерфейси.  
Тобто коли вже існуючий клас не підходить за інтерфейсом, але його потрібно використати.

## Реальні сценарії

### 1. Інтеграція сторонньої бібліотеки  
Наприклад, програма працює з XML, а стороння бібліотека обробляє тільки JSON.  
Адаптер перетворює виклики XML у формат, зрозумілий бібліотеці.

### 2. Підключення старої системи до нової  
Стара система має інший інтерфейс, але її потрібно інтегрувати без переписування коду.

### 3. Робота з різними API  
Коли потрібно уніфікувати роботу з різними сервісами через спільний інтерфейс.

## Що вирішує

- Проблему несумісності інтерфейсів.
- Дозволяє повторно використовувати існуючий код.
- Не змінює клієнтський код.

---

# 2. Facade (Фасад)

## Коли застосовувати

Facade застосовується тоді, коли система має складну підсистему з великою кількістю класів, і потрібно спростити взаємодію з нею.

## Реальні сценарії

### 1. Домашній кінотеатр  
Замість окремого керування проєктором, плеєром і підсилювачем — один метод WatchMovie().

### 2. Платіжна система  
Замість роботи з перевіркою картки, банком, логуванням і безпекою — один метод ProcessPayment().

### 3. Ініціалізація складного модуля  
Наприклад, запуск гри або серверу через один метод Start().

## Що вирішує

- Спрощує клієнтський код.
- Приховує складність підсистеми.
- Зменшує залежність від великої кількості класів.

---

# 3. Proxy (Заступник)

## Коли застосовувати

Proxy використовується, коли потрібно контролювати доступ до об’єкта або додати додаткову логіку без зміни самого об’єкта.

## Реальні сценарії

### 1. Ледаче завантаження зображень  
Об’єкт створюється тільки при першому зверненні.

### 2. Контроль доступу  
Перевірка прав користувача перед виконанням операції.

### 3. Кешування результатів  
Proxy зберігає результат і не виконує повторні дорогі обчислення.

## Що вирішує

- Контроль доступу.
- Оптимізацію ресурсів.
- Додавання логіки без зміни основного класу.

---

# Ключова відмінність між Adapter, Facade та Proxy

- **Adapter** змінює інтерфейс, щоб зробити його сумісним.
- **Facade** спрощує інтерфейс складної системи.
- **Proxy** контролює доступ до об’єкта, не змінюючи його інтерфейс.

---

# Висновок

Adapter, Facade та Proxy вирішують різні проблеми інтеграції та структурування коду.  
Adapter потрібен для сумісності, Facade — для спрощення, Proxy — для контролю та оптимізації доступу.  
Правильний вибір патерна допомагає зробити систему більш гнучкою та зрозумілою.
