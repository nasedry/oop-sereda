## 1. Характеристики анти-патерну God Object 
Анти-патерн God Object —це коли один клас виконує занадто багато задач і стає ніби головним у всій програмі. Такий клас містить багато методів і змінних, працює з різними частинами системи та знає забагато про її внутрішню логіку. Через це код стає складним для читання, тестування і підтримки, а будь-які зміни в одному місці можуть вплинути на всю програму.

## 2. Приклад класу, який порушує SRP
Принцип єдиної відповідальності (SRP) означає, що клас повинен відповідати лише за одну задачу. У наведеному нижче прикладі клас порушує цей принцип, оскільки він одночасно створює користувача, зберігає дані в базу та відправляє повідомлення електронною 
поштою:

class UserManager
{
    public void CreateUser(string name)
    {
        Console.WriteLine($"User {name} created");
    }

    public void SaveToDatabase(string name)
    {
        Console.WriteLine($"Saving {name} to database");
    }

    public void SendEmail(string name)
    {
        Console.WriteLine($"Sending email to {name}");
    }
}
Цей клас має кілька відповідальностей, тому при зміні логіки бази даних або email доведеться змінювати один і той самий клас, що є неправильним підходом.

## 3. Рефакторинг класу для дотримання SRP
Щоб дотримуватися принципу SRP, треба розділити обов’язки між різними класами. Кожен клас повинен виконувати лише одну конкретну функцію. Наприклад, можна створити окремий клас для зберігання даних користувача, окремий клас для роботи з базою даних і окремий клас для відправки повідомлень:
{
class User
{
    public string Name { get; set; }
}

class UserRepository
{
    public void Save(User user)
    {
        Console.WriteLine($"Saving {user.Name} to database");
    }
}

class EmailService
{
    public void Send(User user)
    {
        Console.WriteLine($"Sending email to {user.Name}");
    }
}
}
Після такого рефакторингу кожен клас має одну відповідальність, код стає зрозумілішим, легшим у підтримці та відповідає принципам чистої архітектури.
